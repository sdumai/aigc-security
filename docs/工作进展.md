## 一、界面优化

围绕可读性、一致性与使用体验，对平台前端进行了系统化界面优化，主要包括设计系统建设、页面动效与布局交互修复三方面。

### 1.1 设计系统与视觉规范

**背景**：原有界面依赖零散色值与默认字体，缺乏统一语义，不利于维护与系统一致性；安全/学术类平台需传递稳重、可信的视觉基调。

**方案**：在根目录（ `src/index.css` ）中建立基于 CSS 变量的设计令牌（Design Tokens），并统一应用到布局与各业务页（首页、生成/检测/数据各模块）。通过灵活运用 CSS 变量的方式动态计算生成样式，减少代码量的同时增加主题的可扩展性。

```css
:root {
  --color-primary: #1e3a5f;
  --color-bg-page: #f6f7f9;
  --color-text: #1a202c;
  --font-heading: "Noto Serif SC", serif;
  --font-body: "Noto Sans SC", "PingFang SC", sans-serif;
  --radius-lg: 12px;
  --shadow-md: 0 4px 12px rgba(30, 58, 95, 0.08);
}
```

## 二、生成模块模型接入

本模块通过对接火山引擎视觉智能与方舟大模型等**开放 API**，在平台内复现并验证 Deepfake 相关生成能力与数据流，为后续检测与对比实验提供可控样本来源。能力实现采用「接口对接 + 前后端协议约定」方式，快速验证流程与效果。

### 2.1 Deepfake 人脸生成

#### 2.1.1 人脸替换

**能力与参考**：实现「模板图 + 素材图 → 融合图」的人脸替换流程，能力设计与约束参考火山引擎 [人像融合 3.6](https://www.volcengine.com/docs/86081/1660449)（如人脸尺寸、正脸角度、图片格式与大小等）。当前后端对接视觉智能人脸替换接口，完成双图输入与融合结果返回。

**前后端约定**：前端以 JSON Body 提交 `templateBase64`（目标/模板图）、`imageBase64`（源人脸图），均为 Base64 字符串；服务端转发至视觉接口后，返回 `imageUrl`（data URI 或 URL）及简要 `message`。模板图与素材图顺序与文档一致：先素材图（换脸源），后模板图（被替换场景），避免概念混淆。

```json
// 请求 Body 示例
{ "templateBase64": "<目标图 base64>", "imageBase64": "<源人脸图 base64>" }
// 响应
{ "imageUrl": "data:image/jpeg;base64,...", "message": "人脸替换完成" }
```

**实现要点**：

- 服务端使用 HMAC-SHA256 签名（X-Date、Host、Body 哈希），`Content-Type: application/x-www-form-urlencoded`

  ```js
  function volcHmac(secret, s) {
    return require("crypto").createHmac("sha256", secret).update(s, "utf8").digest();
  }
  ```

- Query 为 `Action=FaceSwap&Version=2020-08-26`；返回融合图 Base64 后由前端以 data URI 展示或落库。

#### 2.1.2 人脸动画

**能力与参考**：基于单张人脸图与文本描述，生成短时驱动动画视频，流程与能力参考 [Seedance 图生视频](https://www.volcengine.com/docs/82379/1366799)（首帧图 + 提示词 → 视频）。采用 **Seedance 1.0 Lite I2V** 模型，通过创建异步任务 + 轮询结果的方式获取视频 URL。

**前后端约定**：前端提交 `imageBase64`（首帧人脸图）、`prompt`（动作/场景描述，可选，默认兜底文案）；服务端创建 `contents/generations/tasks`，content 为 `[{ type: "text", text }, { type: "image_url", image_url: { url: dataUrl }, role: "reference_image" }]`，并轮询直至 `status === "succeeded"`，将 `content.video_url` 作为 `videoUrl` 返回。

```json
// 请求 Body
{ "imageBase64": "<首帧图 base64>", "prompt": "让图中人脸做自然的微笑和轻微点头动作" }
// 响应
{ "videoUrl": "https://...", "message": "人脸动画视频生成成功。" }
```

**实现要点**：鉴权使用 API Key（Bearer），与视觉智能的 AK/SK 相互独立；轮询间隔 3s、上限约 6 分钟，避免长时间阻塞；输出规格固定为 `ratio: "16:9"`、`duration: 5`，可按需扩展为前端可配（后续有需求适配）。

#### 2.1.3 属性编辑

**能力与参考**：基于自然语言指令对给定图片进行编辑（如改背景、改风格、改局部实体、人脸面部特征），能力与接口形态参考火山引擎 [图生图 3.0-指令编辑 SeedEdit 3.0](https://www.volcengine.com/docs/86081/1804561)。该模型为文本引导的扩散式图像编辑，在遵循指令与保留原图细节之间做平衡；

**实现：**接入采用 **visual 网关同步转异步**：提交任务（`CVSync2AsyncSubmitTask`）获取 `task_id`，再轮询查询（`CVSync2AsyncGetResult`）直至 `status === "done"`。

**前后端约定**：前端提交 `prompt`（编辑指令，必填）、`imageUrl` 或 `imageBase64`（待编辑图，二选一）；服务端将图片统一为 `binary_data_base64` 或 `image_urls` 传入 visual，Body 固定 `req_key: "seededit_v3.0"`，可选 `seed`、`scale`（默认 0.5）。结果以 `image_urls` 或 `binary_data_base64` 其一返回，前端统一收 `imageUrl` 展示。

```json
// 请求 Body
{ "prompt": "背景换成演唱会现场", "imageBase64": "<待编辑图 base64>" }
// 响应
{ "imageUrl": "https://... 或 data:image/jpeg;base64,...", "message": "属性编辑完成。" }
```

**鉴权相关**：visual 鉴权使用与人脸融合相同的 AK/SK，签名需包含 `Content-Type`、`X-Content-Sha256` 等（application/json Body）；轮询间隔 2s、最多约 2 分钟；业务错误码（如 50413 文本风险）由服务端解析后以 500 + 文案返回前端，便于用户理解和错误排查。

![image-20260305012209508](/Users/shixu.mai/Library/Application Support/typora-user-images/image-20260305012209508.png)

**鉴权流程**：封装高阶函数，生成 Authorization 鉴权头

> - 规范化请求：用 method、pathName、query、host、x-date 等拼出「规范请求」字符串；有 body 时还会带上 content-type、x-content-sha256（body 的 SHA256）。
> - 凭证与签名字符串：用日期、region、service 等拼出 credentialScope，再和规范请求的哈希一起组成 stringToSign（要签名的内容）。
> - HMAC 派生密钥链：用 secretAccessKey 按「日期 → region → service → "request"」做多级 HMAC-SHA256，得到最终签名密钥。
> - 计算签名并返回：用该密钥对 stringToSign 做 HMAC-SHA256，得到 signature，最后拼成：`HMAC-SHA256 Credential=.../..., SignedHeaders=..., Signature=...`

```js
function volcSign(params) {
  const {
    method = "GET",
    pathName = "/",
    query = {},
    headers = {},
    accessKeyId = "",
    secretAccessKey = "",
    bodySha,
    host = VOLC_CV_HOST,
    service = VOLC_CV_SERVICE,
    region = VOLC_CV_REGION,
    contentType,
  } = params;
  const datetime = headers["X-Date"] || volcGetXDate();
  const date = datetime.substring(0, 8);
  const bodyShaHex = bodySha || volcHash("");
  let signedHeaderKeys, canonicalHeaders;
  if (contentType && bodySha) {
    signedHeaderKeys = "content-type;host;x-content-sha256;x-date";
    canonicalHeaders = `content-type:${contentType}\nhost:${host}\nx-content-sha256:${bodyShaHex}\nx-date:${datetime}\n`;
  } else {
    signedHeaderKeys = "host;x-date";
    canonicalHeaders = `host:${host}\nx-date:${datetime}\n`;
  }
  const canonicalRequest = [
    method.toUpperCase(),
    pathName,
    volcQueryToString(query) || "",
    canonicalHeaders,
    signedHeaderKeys,
    bodyShaHex,
  ].join("\n");
  const credentialScope = [date, region, service, "request"].join("/");
  const stringToSign = ["HMAC-SHA256", datetime, credentialScope, volcHash(canonicalRequest)].join("\n");
  const kDate = volcHmac(secretAccessKey, date);
  const kRegion = volcHmac(kDate, region);
  const kService = volcHmac(kRegion, service);
  const kSigning = volcHmac(kService, "request");
  const signature = volcHmac(kSigning, stringToSign).toString("hex");
  return [
    "HMAC-SHA256",
    `Credential=${accessKeyId}/${credentialScope},`,
    `SignedHeaders=${signedHeaderKeys},`,
    `Signature=${signature}`,
  ].join(" ");
}
```

### 2.2 多模态内容生成

本小节在平台内复现「文本/图像 → 图像/视频」的多模态生成链路，覆盖文生图、文生视频、图生视频三类能力，均通过方舟模型复现，与 2.1 节一致：以接口对接方式验证流程与数据格式，为检测与样本构建提供可控多模态输出。实现参考方舟 [Seedream 4.0–5.0 图像生成](https://www.volcengine.com/docs/82379/1824121) 与 [Seedance 视频生成](https://www.volcengine.com/docs/82379/1366799) 文档。

多模态内容生成在平台内统一走方舟 API Key 鉴权，文生图为同步调用，文生视频与图生视频为异步任务 + 轮询，输出（imageUrl/videoUrl）可供检测模块与数据管理模块复用，形成「生成 → 输出 → 检测」的闭环验证基础。

#### 2.2.1 文生图

**能力：根据自然语言提示词生成单张图像，用于快速产出可用于检测或对比的图片样本。对接方舟 **Seedream 4.0 模型（`doubao-seedream-4-0-250828`），调用服务端 `images/generations` 同步接口，能力与参数形态参考 [Seedream 教程](https://www.volcengine.com/docs/82379/1824121) 中的文生图部分。

**前后端约定**：前端提交 `prompt`（必填）、可选 `size`（默认 `2K`）、`watermark`（是否加水印）；前端请求服务端，服务端转发至方舟后，从 `data[0].url` 取图并封装为 `imageUrl` 与 `message` 返回。

```json
// 请求 Body
{ "prompt": "写实风格，蓝天下的白色雏菊花田", "size": "2K", "watermark": false }
// 响应
{ "imageUrl": "https://...", "message": "图像生成成功！" }
```

**注意**：鉴权使用方舟 API Key（Bearer），与视觉智能 AK/SK 分离；`n: 1`、`response_format: "url"` 固定，便于前端直接展示或落库。支持有无水印选项。

#### 2.2.2 文生视频

**能力**：仅凭文本描述生成短视频片段，流程与能力参考 [Seedance 视频生成](https://www.volcengine.com/docs/82379/1366799)（创建任务 → 轮询直至 `succeeded`）。采用方舟 Seedance 1.0 Lite T2V，通过 `contents/generations/tasks` 创建任务。

**前后端约定**：前端提交 `prompt`（必填）、可选 `ratio`（默认 `16:9`）、`duration`（默认 5 秒）；服务端创建任务后轮询（间隔 3s、上限约 6 分钟），将 `content.video_url` 作为 `videoUrl` 返回。

```json
// 请求 Body

// 响应
{ "videoUrl": "https://...", "message": "视频生成成功！" }
```

**实现要点**：与 2.1.2 人脸动画共用 `pollArkVideoTask` 与方舟 Base URL；失败或超时由服务端统一返回 500 与可读文案。

#### 2.2.3 图生视频

**能力与参考**：基于一张或多张参考图与文本提示，生成与参考风格/主体一致的短视频，对应 Seedance 文档中的「图生视频-基于参考图」场景。采用 **Seedance 1.0 Lite I2V**，content 为 `text` + 多段 `image_url`（`role: "reference_image"`），便于多图融合描述（如「图1的人物 + 图2的场景」）。

**前后端约定**：前端提交 `prompt`（必填）、`imageBase64List`（至少一张图的 Base64 数组）、可选 `ratio`、`duration`；服务端将 Base64 转为 data URL 填入 `image_url.url`，组装 content 后创建任务并轮询，返回 `videoUrl`。

```json
// 请求 Body
{ "prompt": "[图1]的人物坐在[图2]的草坪上，卡通风格", "imageBase64List": ["<base64_1>", "<base64_2>"], "ratio": "16:9", "duration": 5 }
// 响应
{ "videoUrl": "https://...", "message": "视频生成成功！" }
```

**实现要点**：参考图顺序与提示词中的「图1」「图2」对应；方舟对参考图数量与分辨率有限制，前端可做数量与大小校验以降低失败率。

## 三、检测模块模型接入

本模块在平台内复现「图像/视频 → 真伪与安全判定」的检测链路，覆盖**虚假内容检测**（是否 AI 生成/合成）与**敏感内容检测**（是否含违规内容）两类能力，通过火山方舟**多模态理解 API**（图片理解、视频理解）对接：将待检媒体与固定判定提示词一并传入视觉大模型，解析模型输出的结构化 JSON 作为检测结果。实现与接口形态参考 [图片理解](https://www.volcengine.com/docs/82379/1362931) 与 [视频理解](https://www.volcengine.com/docs/82379/1895586) 文档。

### 3.1 虚假内容检测

**能力与参考**：对单张图片或单段视频进行「是否 AI 生成/深度伪造/合成」的二分类判定，并给出置信度与理由，用于对生成模块产出内容做真伪校验。后端调用方舟 **Chat API**（`chat/completions`），使用支持视觉输入的模型接入点（配置项 `VOLC_ARK_VISION_MODEL`，如 doubao-seed 系列等），请求体为「媒体块 + 固定系统提示词」，提示词约束模型仅输出一个 JSON 对象，便于解析。图片形态参考 [图片理解](https://www.volcengine.com/docs/82379/1362931)（`image_url` / Base64），视频形态参考 [视频理解](https://www.volcengine.com/docs/82379/1895586)（`video_url` / Base64）。

**前后端约定**：

- **图片**：前端提交 `imageUrl`（公网 URL）或 `imageBase64`；服务端组装 `messages[].content = [ image_url 块, text 块(判定 prompt ) ]`，从回复中抽取 JSON，归一化为 `isAIGenerated`、`confidence`、`reason` 返回。
- **视频**：前端提交 `videoUrl` 或 `videoBase64`（格式 `data:video/mp4;base64,...`）；服务端同样「视频块 + 判定 prompt」调用 Chat，解析 `is_ai_generated`、`confidence`、`reason` 后封装为 `isFake`、`confidence`、`model`、`details`（含 segmentConclusion、machineLabel 等）返回，与虚假检测页展示结构一致。

```json
// 图片检测请求 Body
{ "imageUrl": "https://..." } 或 { "imageBase64": "<base64>" }
// 图片检测响应（示例）
{ "isAIGenerated": false, "confidence": 0.2, "reason": "未发现明显 AI 生成痕迹" }

// 视频检测请求 Body
{ "videoUrl": "https://..." } 或 { "videoBase64": "<base64>" }
// 视频检测响应（示例）
{ "isFake": false, "confidence": 0.15, "model": "视频 AI 生成识别", "details": { "segmentConclusion": "无 AI 生成痕迹的常规视频", ... } }
```

> **注意**：鉴权使用方舟 API Key（Bearer）；回复中通过正则抽取 `{...}` 再 `JSON.parse`，避免模型多说的自然语言导致解析失败；`temperature` 取 0.1 以稳定输出格式。

### 3.2 敏感内容检测

**能力与参考**：对单张图片或单段视频进行内容安全审核，判定是否含暴力、色情、政治敏感等违规内容，并给出安全建议（pass/review/block）与违规标签。基于方舟 Chat 多模态 API，使用同一视觉模型接入点（`VOLC_ARK_VISION_MODEL`），仅将系统提示词换为「安全审核助手」约束，要求输出 `safe`、`suggestion`、`labels`、`reason`。

**前后端约定**：

- **图片**：前端提交 `imageUrl` 或 `imageBase64`；服务端调用 `volc-ims`，返回 `safe`、`suggestion`、`labels`、`reason`。
- **视频**：前端提交 `videoUrl` 或 `videoBase64`；服务端调用 `volc-video-ims`，返回结构同上。前端据此计算 risk（low/medium/high）、riskScore、suggestions 等用于展示。

```json
// 请求 Body（图片/视频二选一）
{ "imageUrl": "https://..." } 或 { "imageBase64": "..." }
{ "videoUrl": "https://..." } 或 { "videoBase64": "data:video/mp4;base64,..." }
// 响应（统一）
{ "safe": true, "suggestion": "pass", "labels": [], "reason": "未检测到违规内容" }
```

> **注意**：与虚假内容检测共用同一 Chat 端点与模型配置，仅 prompt 不同；视频需满足方舟视频理解的长度与格式限制（如文档所述），Base64 时需带 `data:video/mp4;base64,` 前缀。
